# DiffInk 应用中出现的问题

在我们自己的数据集使用DiffInk的时候，无法进行正常的输出（可见§3），只有训练分布内的输入能成功，且VAE部分未出现问题，想请作者帮判断，是模型的哪个部分出现问题？应该如何进行解决？

**文档结构**：

- §1 讲 **gfont → DiffInk 五元组输入格式**的转换细节（纯格式说明）。
- §2 对比 **三种 style reference 输入来源**（gfont 转、真实手写图 → InkSight、val.h5 直接取样）的数据统计。
- §3 对比三种来源在 DiffInk 上的 **完整推理输出效果**。
- §4 **VAE-only 重建诊断**：单独跑 VAE encode→decode，隔离 VAE 和 diffusion 两部分。
- §5 **selfrecon 自重建 latent 诊断**：让 target = style、dump DiT 前后 latent，把瓶颈进一步锁在 DiT 上。
- §6 汇总想请原作者确认的问题。

## 1. gfont → DiffInk 输入格式的转换

入口函数：

```python
gfont_to_diffink(chars, font_data, char_gap=30.0, target_y_range=5.0)
    -> (point_seq: np.ndarray[T, 5], char_points_idx: list[int])
```

返回的 `point_seq` 和 `char_points_idx` 直接作为 DiffInk handler 的 `style_strokes`（base64 编码后）与 `char_points_idx` 字段送入。

**五步流程**：

1. **解析 gfont**：`.gfont` ZIP 里每个文件按 Unicode 码点命名，读二进制得到 `{char: [stroke_array, ...]}`（每笔 `[Ni, 2]` 浮点点序列）。
2. **逐字预处理**：x 平移到该字左边缘为 0；y 取反（gfont 是屏幕坐标 y 向下 → CASIA 数学坐标 y 向上，由 `visual.py` 渲染器 `y_scaled = (y_max - y) * scale` 反证）。此阶段不缩放。
3. **五元组 `[x, y, is_next, is_new_stroke, is_new_char]`**：笔画内部点 `(1,0,0)`、笔画末（非该字最后一笔）`(0,1,0)`、字符结束标记 `(0,0,1)`。
4. **水平拼接 + 记录边界**：每字后 `x_offset += char_w + char_gap`；`cpi[i] = len(all_points)` = 下一字第一点的索引（**右开切分点**，末字 `cpi[-1] = T`）。格式从 val.h5 真实样本反推对齐（31 字样本 `cpi=[9, 21, ..., 802]`, 总点 802, `cpi[-1]==T`），无额外"过渡点"插入。
5. **全局等比缩放**：`scale = target_y_range / y_range`（`target_y_range=5.0`）同时乘 x 和 y（**不逐字缩放**，保留字间相对尺度），y 居中到 0，最终 y ∈ [−2.5, +2.5]。

**修复记录**：早期版本在结束标记后多插了一个冗余"过渡点"导致 `cpi[-1] = T−1`、边界 flag 变成 `(0,0,1) → (1,0,0) → (1,0,0)`。对比 val.h5 后已修。

- 问题不在模型本身（val.h5 输入能正常工作）；
- 问题也不单纯在"char_points_idx 格式 bug"或"y 缩放"上——即便修完这两处，[1] 和 [2] 依旧糊；
- [1]（gfont 矢量）和 [2]（InkSight 还原）的失败模式高度相似，说明瓶颈更深——大概率是 §2.2 表里其它维度（采样密度、字间距、单字宽度）或未覆盖到的训练侧预处理环节造成的 distribution shift。

## 4. VAE-only 重建诊断

为了把 §3 的失败定位到 pipeline 的哪一层，单独跑一个脚本：**只做 `VAE.encode → decode → GMM sample`，不经过 DiT / diffusion / prefix mask**。

![vae reconstruction comparison](<test_outputs_fixed/vae_reconstruct_comparison.png>)

每组上半 = reference，下半 = VAE 重建：

| # | 输入 | VAE 重建质量 | latent 统计 |
|---|---|---|---|
| [1] | gfont 5-tuple（本文重点） | **几乎完美重建**（天地黄宇宙洪荒严寒[永] 所有字都可辨） | mean=−0.044 std=1.208 min=−6.12 max=5.18 |
| [2] | InkSight 还原的真实手写 | **几乎完美重建**（人生得意须尽[欢] 所有字都可辨） | mean=−0.052 std=1.007 min=−5.66 max=5.71 |
| [3] | val.h5 CASIA 样本 | **几乎完美重建**（31 字完整句子） | mean=−0.093 std=1.268 min=−9.37 max=11.34 |

**结论**：VAE **不是瓶颈**。三种输入（无论训练分布内还是我们新构造的 gfont / InkSight）经 VAE 都能干净重建。这把 §3 的失败定位到 **diffusion 模型 + handler 的条件/掩码策略**上——VAE 的 latent 空间对我们的输入是完整可表达的，但 DiT 在 suffix mask 区域去噪回来的 latent 解不出可读的字。

（唯一可见的统计差异：[3] 的 latent 最大/最小绝对值（11.34 / 9.37）明显比 [1][2] 大——可能因为序列更长，但也可能意味着我们的输入使 latent 分布偏中心化，信息密度不够。）

## 5. selfrecon 自重建 latent 诊断：瓶颈进一步锁在 DiT

§3 证明了输出崩坏，§4 证明 VAE 不是瓶颈——两者合起来把问题指向 DiT + handler 的条件/掩码策略。为了再收紧一步，做一个**极端受控的自重建对照**：让 target 文本和 style 文本**完全相同**，然后 dump DiT 前后的 latent 看它到底在做什么。

### 5.1 实验设置

- `reference_text = target_text = "天地黄宇宙洪荒严寒永"`（同 10 字 gfont 笔迹，402 点）
- `num_style_chars = 10`，所以 `full_text = style_text + target_text` = 20 个完全相同的字
- handler 的常规策略把 `ref_strokes` tile 成 `work_strokes` = 804 点，前 402 点 = 原笔迹，后 402 点 = 原笔迹抄一遍
- **期望**：target 区的 DiT 输出应当和 style 区的 feat 几乎一致——输入笔迹相同、text 条件相同，唯一区别只是 mask 标记哪段要去噪

### 5.2 Pipeline 与 latent 切片命名

```
ref_strokes (402 pts, 10 字)
    │ tile x2
    ▼
work_strokes (804 pts, 20 字, 后半 = 前半复制)
    │
    ▼
VAE encoder → feat [101 × 384]
              ├── feat[0:50]    前 10 字 style 的编码     = feat[style]
              └── feat[50:100]  后 10 字"抄一遍"的编码    = feat[target]
    │
    ▼
DiT (text 条件 = 20 字完整文本, cond_mask 标 target 区需去噪)
    │
    ▼
x_pred [101 × 384]
    ├── x_pred[0:50]    DiT 在 style 位置的输出   = x_pred[style]  (后续被 mask 丢弃)
    └── x_pred[50:100]  DiT 在 target 位置的输出  = x_pred[target] (真正送进 decoder)
    │
    ▼
x_mix = x_pred · mask + feat · (1 - mask)   (target 用 x_pred, style 保留 feat)
    │
    ▼
VAE decoder → 最终笔迹
```

### 5.3 Latent 统计

| 切片 | n | mean | std | 说明 |
|---|---|---|---|---|
| `feat[style]`   | 51 | −0.042 | 1.204 | 真实 gfont 笔迹的 VAE 编码 |
| `feat[target]`  | 50 | −0.039 | 1.234 | tile 复制笔迹的 VAE 编码，**和 feat[style] 几乎一致**（符合预期，tile 出来的内容相同）|
| `x_pred[style]` | 51 | −0.063 | 1.150 | DiT 在 style 位置的输出——最后被 mask 丢弃，仅作 sanity check |
| `x_pred[target]`| 50 | −0.040 | **0.919** | DiT 在 target 位置的输出——真正影响结果，**std 被压到 0.92**（明显收缩）|

**位置-wise L2 差 `‖x_pred − feat‖₂`**（每个位置都是 384 维向量）：

- **style 区（mask=0）**：mean = **7.4** → DiT 几乎把已知 latent passthrough ✓
- **target 区（mask=1）**：mean = **27.8**，峰值 **43** → DiT 把 target latent 拽到离输入很远的位置

### 5.5 结论

这组实验排除了 §3 还未排除的几种可能性：

1. **不是"模型没见过 target 字"**——target 和 style 是同 10 个字。
2. **不是"笔迹 distribution 没见过"**——target 区的 feat 就是 style 笔迹 tile 出来的，和 style 区 feat 统计一致。
3. **不是"VAE 解不出"**——§4 已证，VAE 对同款 latent 能完美解码。
4. **不是"mask 切分错"**——x_pred[style] 几乎 passthrough（L2 差 7.4），说明边界对齐正常。

剩下的唯一解释：**DiT 在 target 区主动把 latent 拉到了一个不是 `feat[target]` 的位置**，且这个新位置 decode 出来的笔迹不成字。即使输入条件"内容完全重复"这种最简单的退化情形，DiT 也做不到 identity 输出。

这指向两个耦合假设：

- **(a) handler 的 tile 填充 ≠ 训练时 target 区的真实编码**。训练时 target 位置的 feat 来自**真实的 target 笔迹**参与 VAE 编码；推理时我们用 ref 笔迹 tile 过来充数，DiT 在 cross-attention 里识别出"target 标记"后按训练目标生成新内容，不会把它当作"已知答案 passthrough"。
- **(b) DiT 的训练目标本身就是"在 target 区生成 ≠ feat[target] 的东西"**——所以即使 selfrecon 情境下 feat[target] 碰巧是"正确答案"，DiT 仍然会按训练惯性把它改掉。

两条假设都指向同一件事：**推理侧 handler 的 suffix 填充策略不等价于训练侧的 target 输入**，而 DiT 对这个差异非常敏感。

## 6. 对模型算法的疑问

在对VAE进行训练的部分，是
